## 1) 필리핀 대표님 (2026-07 · 주간사역보고서 기준)

### 현재 관리 방식
- **담당자별 + 업무별**로 나누어 관리, **주간사역보고서**로 취합
- 보고서 구조: `구분(업무) · 이번 주 진행사항 · 담당자 · 다음 주 사역계획` + 방향성/월별 목표 + 특이사항
- 업무(구분) 예시: 지원 사역 / M.K 사역 / 전략선교(온택+필잇) / RCS(레스토랑·진행) / 유통·차량 / 단기선교

### 지부 조직 구조 (계층형)
- **교육사역**: 학사일정 · 학생관리 · 수업관리 · 교사관리 · 교육행정 · 개발
- **엔츠**: 위 + 지역교회
- **전략선교팀**: 온택(세부영역) · 선교지원(세부영역) · 케이푸드(세부영역)
  → 즉 **대분류 > 세부영역** 2단계 구조

### 핵심 문제 (대표님 + 사용자 진단)
- 선교사님들이 **보고서 작성에 시간이 많이 걸림**
- **"보고를 위한 보고서"**가 되어버림 (실제 일보다 보고가 목적이 되는 문제)
- 보고서 자체가 목적이 아니라, **일이 잘 되게** 하는 게 목적이어야 함

### 보고서 안 "발전 및 개발영역"에 이미 적힌 방향
- 한 번에 전체 상황을 알 수 있게 기록
- 다음 주 보고서는 전 주 토요일 저녁까지 준비
- (영어 공부 — 개인 목표)

## 3) 사용자(총괄)의 궁극 목표
- **선교사님**: 이 프로그램을 실제로 쓰면서 효과를 봄 → "이거 쓰니 **까먹거나 딜레이되는 게 최소화**됐다"
- **디렉터**: 각 선교사가 **자기 업무로 작성한 것을 그대로 열람** → "이렇게 진행 중이군" 파악 → **조절 · 업무 재분배**
- 디렉터는 **직관적으로** 전체를 봄
- 내용은 **지부마다 조금씩 달라도** 됨 (구조는 공통, 항목은 유연)

## 4) 설계 원칙 (위 문제의 해법)
> **보고서는 따로 쓰는 문서가 아니라, 매일의 업무 관리에서 자동으로 나오는 "부산물"이어야 한다.**

- 선교사는 **자기 일을 위해** 할일을 관리(오늘/이번주 + 밀림/주기) → 까먹음·딜레이 최소화
- 그 데이터로 **주간보고서를 자동 생성** (이번 주 한 일 = 완료·밀림, 다음 주 계획 = 다음주 할일)
- 디렉터는 **같은 데이터를 열람**만 → 별도 취합 불필요

### 보고서 ↔ 프로그램 매핑
| 주간보고서 | 프로그램 |
|---|---|
| 담당자 | 선교사 계정 |
| 구분(업무) | 업무 카테고리 (대분류 > 세부영역) |
| 이번 주 진행사항 | 이번 주 완료/진행 할일 (자동 집계) |
| 다음 주 사역계획 | 다음 주 계획 할일 |
| 상반기/월 목표 | 상위 목표(카테고리별) |
| 특이사항 | 메모 |

## 5) 열린 결정 사항 (한국 본부 피드백 후 확정)
1. **데이터 공유 방식**: 디렉터가 여러 선교사 데이터를 보려면 → **공유 서버 필요**(localStorage 불가). 이미 세팅해 둔 **Supabase** 기반으로 갈지 결정.
2. **주기(cadence)**: 필리핀은 **주간** 보고. 프로그램 기본 리듬을 주간 중심으로 할지.
3. **카테고리 계층**: 대분류>세부영역 2단계를 지부별로 편집 가능하게.
4. **권한(역할)**: 선교사 / 디렉터(열람·재분배) 역할 구분.
